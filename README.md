# Local-First LLM Knowledge Base

A personal, local-first research pipeline: articles are captured, ingested, synthesized by a local LLM (Ollama), confidence-scored, and organized into a topic graph viewable in Obsidian. No cloud. No external APIs. Everything runs on your machine.

---

## What This Project Is

This project is built around a few principles:

- **Local-first** — the knowledge base lives in ordinary files on disk.
- **Strong data lineage** — every compiled or generated artifact points back to its inputs.
- **Git-friendly** — plain text files, stable paths, and readable diffs.
- **Obsidian-friendly** — markdown files, folders, and wikilinks work as a browseable vault.
- **No overwriting of raw content** — generated artifacts must never replace source truth.
- **Explicit synthesis** — auto-synthesis is blocked; every synthesis run requires an intentional command.

## Top-Level Folder Structure

### `raw/`

The source-truth layer. All ingested material lands here. Nothing in `raw/` is overwritten by automation.

- `raw/inbox/` — temporary staging area for incoming files (watched by the inbox watcher)
  - `raw/inbox/browser/` — browser-saved HTML files
  - `raw/inbox/clipboard/` — clipboard-captured text
  - `raw/inbox/feeds/` — RSS/Atom feed items
  - `raw/inbox/pdf-drop/` — PDF files
- `raw/articles/` — normalized raw notes derived from articles, blog posts, or webpages
- `raw/notes/` — manual research notes or analyst notes
- `raw/pdfs/` — PDF files or PDF-related source notes
- `raw/archive/` — originals preserved after ingestion (timestamped)

### `compiled/`

The synthesis layer. Notes here combine, summarize, or reorganize raw material into reusable knowledge. All compiled notes must point back to the raw notes they were derived from.

- `compiled/topics/` — topic notes aggregating multiple source summaries (the canonical graph nodes)
- `compiled/source_summaries/` — per-article LLM synthesis (intermediate artifacts)
- `compiled/concepts/` — concept-centric notes (planned Phase 8)
- `compiled/index.md` — auto-generated wiki index with one-line summaries per note

### `outputs/`

The artifact layer. Generated deliverables — answers, reports, lint results.

- `outputs/answers/` — Q&A artifacts produced by `query.py`
- `outputs/reports/` — report-style deliverables and lint output

### `metadata/`

Lightweight structured metadata. No database.

- `metadata/source-manifest.json` — index of every ingested source
- `metadata/review-queue.json` — queue of items awaiting synthesis or approval
- `metadata/review-queue.md` — human-readable queue table
- `metadata/topic-registry.json` — canonical topic slugs, titles, and aliases
- `metadata/feeds.json` — subscribed RSS/Atom feed URLs
- `metadata/prompts/` — prompt-packs generated by `compile_notes.py`

### `scripts/`

Python utilities for every pipeline stage. All scripts use only the Python standard library unless noted.

### `templates/`

Reusable markdown templates for raw notes, compiled notes, and outputs.

---

## Lineage Rules

These rules are the core of the repository:

1. Raw notes are source truth.
2. Compiled notes must point back to raw notes.
3. Outputs must record the query or prompt and the sources used.
4. Generated content must not overwrite raw content.

## Scripts Reference

### `stage_to_inbox.py` — Capture

Routes captured content into `raw/inbox/` before the watcher picks it up.

```bash
# Browser-saved HTML or markdown file
python3 scripts/stage_to_inbox.py browser \
  --input-file ~/Downloads/article.html \
  --title "Article Title" \
  --canonical-url "https://example.com/article"

# Clipboard text
python3 scripts/stage_to_inbox.py clipboard \
  --title "My Note" \
  --text "Content to capture"

# Feed item JSON payload
python3 scripts/stage_to_inbox.py feeds \
  --text '{"title":"Feed Item","url":"https://example.com","content":"..."}'

# PDF drop
python3 scripts/stage_to_inbox.py pdf-drop \
  --input-file ~/Downloads/paper.pdf
```

When staging an `.html` file, markup is stripped automatically before the note body is written. Script tags, style blocks, SVG, and `<head>` content are discarded; block-level elements become line breaks; HTML entities are unescaped.

---

### `inbox_watcher.py` — Ingest, Validate, Queue

Watches `raw/inbox/` for new files and runs the two-stage gate: ingest → validate → queue.

```bash
# Start watcher in foreground (Ctrl-C to stop)
python3 scripts/inbox_watcher.py

# Process whatever is in inbox right now then exit
python3 scripts/inbox_watcher.py --once

# Custom poll interval
python3 scripts/inbox_watcher.py --interval 10
```

For each new file the watcher:

1. Derives title, source type, origin, and canonical URL from the file
2. Calls `ingest.py` to build the normalized raw note and update `source-manifest.json`
3. Validates the note against the required frontmatter and body schema
4. Appends an entry to `metadata/review-queue.json` and `metadata/review-queue.md`
5. Archives the original inbox file to `raw/archive/` with a timestamp

State is tracked in `metadata/.watcher-state.json` (keyed by path + mtime) so restarts never re-ingest the same file.

Runs as a systemd user service: `kb-inbox-watcher.service`

---

### `feed_poller.py` — RSS/Atom Feed Polling

Reads subscribed feed URLs from `metadata/feeds.json`, fetches new items on a schedule, and drops them into `raw/inbox/feeds/`.

```bash
python3 scripts/feed_poller.py
python3 scripts/feed_poller.py --once
```

Runs as a systemd user service: `kb-feed-poller.service`

---

### `synthesize.py` — Synthesis Entry Point

The main synthesis command. Looks up a queued item, generates a prompt-pack, runs Ollama, applies the result, scores it, and aggregates it into the topic note.

```bash
# List all items pending review
python3 scripts/synthesize.py --list

# Synthesize one item by source ID
python3 scripts/synthesize.py SRC-20260412-0001

# Override the compiled note title
python3 scripts/synthesize.py SRC-20260412-0001 --title "My Custom Title"

# Synthesize all pending items in one pass
python3 scripts/synthesize.py --all

# Force overwrite if compiled note already exists
python3 scripts/synthesize.py SRC-20260412-0001 --force

# Use a different Ollama model (default: qwen2.5:14b)
python3 scripts/synthesize.py --all --model qwen2.5:7b
```

Pipeline per item:

1. `compile_notes.py` (prompt-pack mode) → `metadata/prompts/compile-<slug>-synthesis.md`
2. `llm_driver.py` (Ollama streaming) → `tmp/synthesis-output.md`
3. `apply_synthesis.py` → `compiled/source_summaries/<slug>-synthesis.md`
4. `score_synthesis.py` → confidence score + auto-approve if ≥ 0.85
5. `topic_aggregator.py` → `compiled/topics/<slug>.md` updated or created

If Ollama is unreachable, falls back to a scaffold template. Scoring and topic aggregation are non-blocking — failures produce a warning but never abort synthesis.

Queue status flow: `pending_review` → `synthesized` | `synthesis_failed`

---

### `score_synthesis.py` — Confidence Scoring

Runs a self-critique pass through Ollama and assigns a confidence score (0.0–1.0).

| Band | Range | Action |
|------|-------|--------|
| high | ≥ 0.85 | auto-approved |
| medium | 0.65–0.84 | surfaces for human review |
| low | < 0.65 | surfaces for human review |

```bash
python3 scripts/score_synthesis.py SRC-20260412-0001
python3 scripts/score_synthesis.py --all
python3 scripts/score_synthesis.py --list
```

Called automatically by `synthesize.py`. Can also be run standalone.

---

### `review.py` — Human Review

Surfaces synthesized items for human triage, sorted low-confidence first.

```bash
# Show all synthesized items pending review
python3 scripts/review.py list

# Approve one item
python3 scripts/review.py approve SRC-20260412-0001

# Batch-approve all items with confidence ≥ 0.85
python3 scripts/review.py approve --all-high-confidence

# Reject with optional reason
python3 scripts/review.py reject SRC-20260412-0001 --reason "off-topic"
```

---

### `topic_aggregator.py` — Topic Aggregation

Classifies synthesized source summaries against `metadata/topic-registry.json` and merges them into the canonical topic note in `compiled/topics/`.

Matching uses normalized alias/keyword matching against the article title and first 2,000 characters of the body. If no topic matches, the source summary stands alone (same behavior as before Phase 5).

```bash
# Aggregate one source summary
python3 scripts/topic_aggregator.py \
  --topic openclaw-security \
  --source-summary compiled/source_summaries/my-article-synthesis.md \
  --source-id SRC-20260412-0001

# Aggregate all synthesized items that match a topic
python3 scripts/topic_aggregator.py --all

# Show which source summaries are linked to which topics
python3 scripts/topic_aggregator.py --list
```

Called automatically by `synthesize.py` after scoring.

**Adding a new topic:** Add an entry to `metadata/topic-registry.json` with a slug, title, and aliases. Future syntheses will automatically route matching articles into it.

---

### `compile_notes.py` — Prompt-Pack and Scaffold Generation

Takes one or more raw notes and generates a prompt-pack (for LLM synthesis) and/or a scaffold (human-fillable template).

```bash
# Generate a prompt-pack
python3 scripts/compile_notes.py \
  --sources raw/articles/my-article.md \
  --title "My Topic" \
  --source-summary \
  --mode prompt-pack

# Generate both scaffold and prompt-pack
python3 scripts/compile_notes.py \
  --sources raw/articles/a.md raw/articles/b.md \
  --title "Combined Topic" \
  --topic \
  --mode both

# Force overwrite
python3 scripts/compile_notes.py ... --force
```

Categories: `--source-summary` → `compiled/source_summaries/`, `--concept` → `compiled/concepts/`, `--topic` → `compiled/topics/`

---

### `llm_driver.py` — Direct Ollama Synthesis

Reads a prompt-pack, streams Ollama output to the terminal, and passes it through `apply_synthesis.py`. Used directly when you want to synthesize outside the review queue.

```bash
python3 scripts/llm_driver.py \
  --prompt-pack metadata/prompts/compile-my-topic.md \
  --output-type compiled

# Use a different model
python3 scripts/llm_driver.py \
  --prompt-pack metadata/prompts/compile-my-topic.md \
  --model qwen2.5:7b

# Overwrite existing
python3 scripts/llm_driver.py \
  --prompt-pack metadata/prompts/compile-my-topic.md \
  --force
```

Default model: `qwen2.5:14b`. Connects to Ollama at `http://localhost:11434`.

---

### `apply_synthesis.py` — Apply Synthesized Output

Takes a prompt-pack and synthesized markdown (from a file or CLI text) and writes a durable artifact. Sanitizes LLM output before writing (removes citation junk, malformed fences, shell fragments, path-like tokens).

```bash
python3 scripts/apply_synthesis.py \
  --prompt-pack metadata/prompts/compile-my-topic.md \
  --synthesized-file ./tmp/synthesized-note.md

python3 scripts/apply_synthesis.py \
  --prompt-pack metadata/prompts/compile-my-topic.md \
  --output-type answer \
  --text "# Question\n\n...\n\n# Answer\n\n..."
```

---

### `index_notes.py` — Wiki Index

Generates `compiled/index.md` — a one-line-per-note index grouped by category (topics first, then source summaries). Used by `query.py` as a wiki map in every prompt.

```bash
python3 scripts/index_notes.py
python3 scripts/index_notes.py --dry-run
python3 scripts/index_notes.py --json
```

Run after any synthesis or topic aggregation to keep the index current.

---

### `query.py` — Q&A

Loads compiled notes, builds a prompt with the question and wiki context, streams the Ollama response, and files the answer in `outputs/answers/`.

```bash
python3 scripts/query.py --question "What are the best practices for OpenClaw security?"

# Limit context to top 5 most relevant notes (BM25)
python3 scripts/query.py --question "..." --top-n 5

# Preview prompt without calling the model
python3 scripts/query.py --question "..." --dry-run

# Overwrite existing answer
python3 scripts/query.py --question "..." --force
```

Without `--top-n`, all compiled notes are included up to the model's token budget.

---

### `search.py` — BM25 Full-Text Search

Indexes all compiled notes using the Okapi BM25 algorithm. Title matches are boosted 3x. Also powers `--top-n` retrieval in `query.py`.

```bash
python3 scripts/search.py "openclaw security hardening"
python3 scripts/search.py "openclaw security" --top-n 3
python3 scripts/search.py "openclaw" --include-raw
python3 scripts/search.py "openclaw security" --json
```

---

### `lint.py` — Health Checks

Scans the wiki for structural problems. Pure-Python checks run without Ollama; LLM-assisted checks require `--llm`.

**Pure checks:**
- `wikilinks` — dangling `[[wikilinks]]` pointing to files that don't exist
- `orphans` — raw notes not referenced by any compiled note's `compiled_from:` field

**LLM-assisted checks:**
- `coverage` — topic gap analysis via Ollama

```bash
python3 scripts/lint.py
python3 scripts/lint.py --llm
python3 scripts/lint.py --report          # file to outputs/reports/
python3 scripts/lint.py --check wikilinks
python3 scripts/lint.py --dry-run
```

---

### `ingest.py` — Raw Note Normalization

Called by `inbox_watcher.py`. Normalizes source input into a standardized raw markdown note, adds frontmatter, chooses the destination folder, and updates `metadata/source-manifest.json`.

Can also be run standalone:

```bash
python3 scripts/ingest.py \
  --input-file ./inbox/example.txt \
  --title "Example Note" \
  --source-type note \
  --origin local-file
```

## How Obsidian Fits In

Open the repository folder as an Obsidian vault. Folder structure, wikilinks, and YAML frontmatter all work natively. The Obsidian graph surfaces the topic hierarchy automatically:

- `compiled/index.md` — the graph's root node
- `compiled/topics/` — canonical topic nodes linked from the index
- `compiled/source_summaries/` — per-article synthesis, linked from topic notes
- `raw/articles/` — raw articles, linked from source summaries

## How Git Fits In

Because everything is stored as ordinary markdown files, Git tracks the full evolution of the knowledge base — which articles were ingested, when topic notes changed, and how synthesis quality improved over time. Keep commits small and meaningful; avoid force-rewriting source-truth material.

## Obsidian Dataview Queries

Phase 7 adds `approved`, `confidence_score`, and `date_updated` fields to all compiled notes, making them queryable as a live database in [Obsidian Dataview](https://blacksmithgu.github.io/obsidian-dataview/).

```dataview
// All source summaries sorted by confidence (lowest first — review targets)
TABLE confidence_score, approved, date_compiled
FROM "compiled/source_summaries"
SORT confidence_score ASC
```

```dataview
// Items still awaiting approval
TABLE confidence_score, date_compiled
FROM "compiled/source_summaries"
WHERE approved = false
SORT date_compiled DESC
```

```dataview
// Low-confidence notes (below 0.65) across all compiled output
TABLE confidence_score, note_type
FROM "compiled"
WHERE confidence_score < 0.65
```

```dataview
// Recently updated topic notes
TABLE compiled_from, date_updated
FROM "compiled/topics"
SORT date_updated DESC
```

```dataview
// All notes by type and approval state
TABLE note_type, approved, date_compiled
FROM "compiled"
SORT date_compiled DESC
```

## Roadmap

### ✅ Phase 1 — Automated Ingestion

Source adapters (`stage_to_inbox.py`) for browser HTML, clipboard, RSS feeds, and PDF. Inbox watcher (`inbox_watcher.py`) runs a two-stage gate: ingest → validate → queue. HTML-to-text stripping prevents markup from polluting note bodies. Processed-file state keyed by path + mtime so renamed duplicates are never skipped.

**Delivered:** A new document goes from capture to validated-and-queued with two commands.

---

### ✅ Phase 2 — On-Demand Synthesis

`synthesize.py` as a single entry point for queue-driven synthesis. Pipeline: `compile_notes.py` → `llm_driver.py` (Ollama streaming) → `apply_synthesis.py` → queue status update. Scaffold fallback when Ollama is unavailable. `--all` flag processes every pending item in one pass. Compiled notes named with `-synthesis` suffix to prevent Obsidian wikilink ambiguity.

**Delivered:** `python3 scripts/synthesize.py --all` runs the full pipeline end-to-end.

---

### ✅ Phase 3 — Feed Poller & Daemon

`feed_poller.py` fetches new items from subscribed RSS/Atom feeds and drops them into the inbox automatically. Both the inbox watcher and feed poller packaged as systemd user services. Bug fixes: filename collision handling, HTML title extraction, truncated LLM fence stripping.

**Delivered:** After a one-time `systemctl --user enable`, new browser saves and feed items appear in the review queue with zero terminal interaction.

---

### ✅ Phase 4 — Review UX & Confidence Scoring

`score_synthesis.py` runs an Ollama self-critique pass assigning a 0.0–1.0 confidence score. Items scoring ≥ 0.85 are auto-approved. `review.py` CLI surfaces lower-confidence items sorted low-first with `list`, `approve`, and `reject` subcommands. Scoring triggered automatically after every successful synthesis. Queue schema extended with confidence and review fields.

**Delivered:** High-confidence items require zero human action. Low-confidence items surface for quick triage.

---

### ✅ Phase 5 — Topic Aggregation

`topic_aggregator.py` classifies each synthesized source summary against `metadata/topic-registry.json` using normalized alias/keyword matching, then creates or updates a canonical topic note in `compiled/topics/`. The Obsidian graph shows Index → Topic → Source Summaries → Raw Articles. New articles on the same topic expand the topic note rather than creating a new graph node. Called automatically by `synthesize.py` after scoring.

**Delivered:** 459 tests passing. Topic notes grow incrementally as new articles arrive.

---

### Phase 6 — Lint Operation

Extend `lint.py` with LLM-assisted checks beyond the current wikilink and orphan checks:

- **Contradictions** — claims in one source summary that conflict with another on the same topic
- **Missing concepts** — terms referenced across notes with no dedicated page
- **Stale claims** — assertions in topic notes superseded by newer source summaries
- **Low-confidence orphans** — items with confidence < 0.85 never manually reviewed

Add `--fix` flag to auto-create stub pages for missing concepts. Wire into systemd as an optional weekly scheduled job. Output to `outputs/lint-YYYY-MM-DD.md`.

---

### Phase 7 — YAML Frontmatter + Obsidian Dataview

Define a frontmatter schema for each page type (`source-summary`, `topic`, `concept`, `entity`, `comparison`) including `confidence`, `approved`, `sources`, `related`, `created`, and `updated` fields. Update `synthesize.py` and `topic_aggregator.py` to write compliant frontmatter. Backfill existing notes. Add Dataview query examples to this README.

**Unlocks:** Querying the knowledge base as a database from within Obsidian.

---

### Phase 8 — Concept and Entity Page Types

Two new page categories: `compiled/concepts/` and `compiled/entities/`. Update `synthesize.py` to extract candidate concepts and entities from each source and create stub pages. Add `[[wikilink]]` references from topic and source notes into concept/entity pages. Add `--concepts-only` flag for extraction without re-synthesis.

---

### Phase 9 — Operation Log

`compiled/log.md` as an append-only audit trail. `synthesize.py`, `review.py`, and `lint.py` all append structured entries. `log.py` CLI for `tail` and `since YYYY-MM-DD` views. Answers: "what changed in the `openclaw-security` topic note this week?"

---

### Phase 10 — Git Integration

Auto-commit after each synthesis run and review action with descriptive messages:

```
ingest: SRC-20260412-0001 openclaw-security (confidence 0.91)
review: approved SRC-20260412-0001
```

`--no-commit` flag for dry-run/testing. Confirms `.gitignore` excludes inbox staging and scratch files.

---

### Phase 11 — Local Dashboard

Lightweight local web UI (FastAPI + vanilla JS, `localhost:7842`) exposing the full pipeline state without terminal interaction. Sections: Queue Panel (approve/reject inline), Topic Graph Health, Pipeline Status, Ingestion Log, Search. Obsidian deep-link integration via `obsidian://open?path=...` URIs. Packaged as `kb-dashboard.service`.

**Recommended build order:** Phase 6 → 7 → 9 → 10 → 8 → 11
